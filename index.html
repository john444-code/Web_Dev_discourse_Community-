The Big Picture

So, what actually is web development?

Most people think web development is about code. It is — but that's not the whole story. Web development is a discourse community: a group of people united by shared goals, shared language, and shared ways of communicating. Knowing that changes everything about how you enter it.

A discourse community isn't just people who do similar work. According to writing scholar John Swales, it's a group with common goals, specific communication channels, and shared genres — ways of writing that the group has developed together over time.

In web development, those genres are things like README files, GitHub commit messages, pull request comments, and technical documentation. Learning to read and write these isn't optional — it's how you signal that you belong.

The good news: this community was built to be open. Its knowledge is mostly free, online, and community-maintained. You don't need to pay tuition to access it.

"Code may be the product of web development, but writing — in its many technical and collaborative forms — is the medium through which the community sustains itself."

— Primary Research, ENG 102 Survey Study

Pathways In

Three ways people actually break in

There's no single path into web development — and that's one of the things that makes it uniquely accessible. Based on survey data from 15 early-career developers, here are the three dominant entry patterns.

01

Self-directed learning

The majority of developers — 67% — started on their own, using free tools like YouTube tutorials, freeCodeCamp, The Odin Project, and MDN Web Docs. This isn't a workaround; it's the norm. The community has built its own onboarding infrastructure, and it's available to anyone with an internet connection. Starting here means you're already participating in how the community transmits its knowledge.

02

Building real projects

Technical skills solidify through practice, not passive study. Survey respondents overwhelmingly pointed to portfolio projects, collaborative work, and debugging real problems as the thing that actually built their competence. You don't need permission to start — pick a project that interests you and build it. Competence in this community is demonstrated through work that others can see and evaluate.

03

Participating in community spaces

Stack Overflow, GitHub, Discord servers, and Reddit communities like r/webdev are where developers ask questions, share work, and review each other's code. Lurking counts. Reading how experienced developers communicate — how they ask questions, how they write documentation, how they respond to each other — is how you absorb the community's norms before you're fully in it. Respondents who participated in collaborative environments consistently reported stronger confidence than those who learned entirely alone.

"When a newcomer reads through MDN Web Docs to learn HTML, they are not merely absorbing information — they are being socialized into the community's genres, values, and communication norms."
— Survey Research, Building the Web (2026)

The Part Nobody Talks About

Web developers write — a lot.

73% of surveyed developers said writing is a necessary part of their work. The genres they named were consistent across respondents. These aren't optional extras — they're the shared genres that form the backbone of this community.

📄

README files

The first thing anyone reads when they open a project. A good README explains what it does, how to run it, and how to contribute.

💬

Commit messages

Short messages that explain what changed and why. Written hundreds of times across a project's life. Follow community conventions.

🔍

Code comments

Inline writing inside code that explains complex logic to future readers — including your future self. For humans, not machines.

📋

Issue threads

Bug reports and feature requests on GitHub. Good issue writing is a skill: clear, specific, and reproducible so someone can help.

📝

Technical docs

Longer documentation explaining how a system works. GitHub's research shows documentation quality predicts open-source project success.

👥

Pull request reviews

Written feedback on a teammate's code. Requires precision, tact, and shared understanding of what "good" means in this community.

Where to Start

Free resources to get you in the door

The web development community runs on open knowledge. Here are the platforms survey respondents mentioned most — all free, all community-maintained, all available to you right now.

Learn

The Odin Project

A full, community-built curriculum covering HTML, CSS, JavaScript, and more. Free, open-source, and structured like a real learning path rather than isolated videos.

Learn

freeCodeCamp

Free certifications in web development, JavaScript algorithms, and more. Used by a large portion of self-taught developers as their primary starting point.

Reference

MDN Web Docs

The authoritative documentation for HTML, CSS, and JavaScript. Reading MDN regularly is how developers socialize into the community's technical language and norms.

Community

GitHub

Where the community stores its code, documents its projects, and reviews each other's work. Creating an account and exploring open-source repos is itself an entry point into the community.

Community

Stack Overflow

The community's collective memory. Developers post problems; others answer. Reading how questions are asked and answered teaches you the community's communicative norms faster than any tutorial.

Rhetorical Reflection

Why I made this — and how.

For this project, I chose to present my research on the web development discourse community as a website. That choice was deliberate and layered: the medium is the message. A website about web development, built using the same tools and conventions the community uses, reinforces the argument the content is making — that web development is a communicative, rhetorical community, not just a technical one. The form participates in the argument.

My intended audience is undecided ASU students who are considering web development but have no technical background and no clear pathway in. These are students who may assume that coding requires a CS degree, a certain kind of intelligence, or years of structured training. My goal was to disrupt that assumption and replace it with a more accurate picture: that the web development community is largely self-taught, open-access, and built around shared communicative practices that anyone can begin learning. The audience shapes every decision I made — the tone is accessible and direct rather than technical or academic, the structure moves from "what is this?" to "how do I get in?" to "where do I start?", and the visual design is clean and professional without being intimidating.

The purpose of this project is two-fold: to inform and to persuade. I want undecided students to understand what web development actually is — a discourse community, not just a set of coding skills — and I want them to feel that entry is genuinely possible for them. The research finding that 67% of developers entered through self-directed learning is not just a data point; it is the most persuasive thing I can offer this audience. I foregrounded it in the statistics section and returned to it in the resources section to reinforce the message: the community has already built the infrastructure to welcome you. You just need to know it exists.

The exigency for this project is real. Undecided students face major declaration pressure early in their ASU careers, and many rule out fields like web development based on misconceptions — that it requires a formal CS degree, that it's inaccessible without a technical background, or that it's purely a solitary, technical pursuit. My research directly challenges those assumptions, and presenting it in a format targeted at that specific audience makes the research actionable rather than merely academic.

The genre of a website was appropriate for several reasons. Websites allow for structured navigation, visual hierarchy, and the chunking of information in ways that a linear document cannot. They also align with how this particular audience — younger students — already consumes information. A brochure or academic paper would not have reached them with the same immediacy. Additionally, a website gave me the opportunity to make rhetorical decisions about design: the color palette uses ASU's maroon and gold to signal institutional credibility and connection to the student audience; the typography balances a serif display font for warmth with a clean sans-serif for readability; the layout progresses logically from concept to action.

The context informing this project includes the broader cultural moment around career uncertainty and the increasing visibility of self-taught developers in the tech industry. Web development is one of the few high-earning fields where formal credentials are genuinely optional, and that matters for ASU students navigating questions about whether their college investment pays off. This project meets that moment directly.
